부양 가족(인적 공제) 등록 
[작동 원리]
1. DB 1:N(일대다) 관계 매핑
== yearTax - dependent == 1:N 으로 구성 = 하나의 메인 정산 내역에 여러 개의 부양가족 내역이 추가되는 구조
== 이 때 각 dependent 데이터에는 해당 데이터가 어떤 YEAR_TAX에 속해 있는지 식별하기 위해 연결 역할을 하는 FK(외래키) 구조
2. 인공지능 자동 합산 로직
== 부양가족 한명 당 YEAR_TAX의 예상 환급액에 150만원을 자동으로 합산하는 로직이 적용되어 있음
== 실제 연말정산 과정에서 적용되는 1인당 150만원의 인적공제(기본공제) 규칙을 시스템에 그대로 구현한 것입니다.

국세청 PDF 파일 업로드
[작동원리]
1. 파일 업로드 및 데이터 파싱의 비동기 분리
-- 사용자가 다수의 페이지 및 국세청 PDF 자료를 업로드할 때 발생하는 서버 부하가 대기 시간을 방지하기 위해
-> 파일 수신 작업과 데이터 파싱(추출) 작업을 분리
@Async 어노테이션이나 메시지 큐를 활용하여 연산이 무거운 파싱 로직을 메인 스레드가 아닌 백그라운드 스레드에 할당해 처리
2. 고유 식별자(UUID) 기반 데이터베이스 상태 추적
-- 해당 PDF 파일의 난수로 구성된 고유 접수 번호(UUID, Job Id)를 생성
-> 생성된 UUID를 기본키(PK)로 사용하여 DB에 현재 처리 상태를 PROCESSING으로 기록한 뒤, 스레드에 작업을 넘기고 프론트엔드에는 이 UUID 값만 즉시 반환
-> 실제 PDF 파싱 로직이 끝나면, DB의 해당 UUID 값을 COMPLETED로 수정함
3. Ajax 폴링을 통한 실시간 렌더링
-- 프론트엔드에서 최초 응답 받는 UUID를 들고 일정 주기에 서버에 현재 처리 상태를 묻는 Ajax 폴링 요청을 전달
-- 서버의 상태 확인 API는 DB를 조회해 상태 값을 전달 -> 상태가 COMPLETED로 확인 되면 페이지 새로고침 없이 화면에 즉시 출력(동적 렌더링)
4. 정산 데이터 일괄 업데이트
-- PDF 파일 검증 및 파싱이 완료 -> 추출된 세부 공제 데이터를 바탕으로 YEAR_TAX의 RESULT_AMOUNT 수치를 일괄 갱신하여 정산 로직을 수행


JWT 를 활용한 로그인/회원가입 기능
[핵심 컴포넌트]
-- JwtTokenProvider : 토큰 생성, 토큰 검증, 토큰 정보 전달 -> 토큰 제작 및 감별사
-- JwtAuthenticationFilter : 클라이언트 요청 시 -> 헤더의 토큰 정보를 낚아 채서 검증하는 것
-- SecurityConfig : 권한 부여 

JWT 작동 원리

사용자가 입력한 ID/PWD 를 JSON 형태로 변환해서 서버의 로그인 API로 전송
-> 서버 : 수신한 ID/PWD를 DB의 사용자 정보와 대조해서 유효성 검증
-> 검증에 성공하면 서버가 가진 비밀 키를 사용하여 해당 사용자의 식별자와 만료시간이 포함된 JWT를 생성(이 때 서명 알고리즘인 HS256 사용)
-> 서버 : 생성된 문자열 형태의 JWT를 HTTP 응답 본문(RequestBody)에 담아 클라이언트에게 반환
-> 클라이언트 : 응답받은 토큰을 브라우저 내장 저장소인 localStorage에 저장
-> 클라이언트 : 보안이 필요한 데이터를 서버에 요청할 때, localStorage에서 토큰을 읽어옴
-> 이 토큰을 HTTP 헤더에 아래와 같은 규격으로 포함시켜 전송(Authorization: Bearer <토큰문자열>)
-> 서버 : 클라이언트의 요청이 컨트롤러에 도달하기 전, 필터가 요청을 가로 채서 헤더에서 토큰을 추출
-> 서버 : 서버의 비밀키를 사용하여 토큰의 서명이 유효한 지, 데이터가 위조되지 않았는지, 유효기간이 지나지 않았는지 검증
-> 서버 : 토큰 검증을 통과하면(인증 성공), 서버는 해당 요청을 정상적으로 처리하여, 클라이언트에게 대시보드 데이터를 응답(200, OK)합니다. 
-> 만약 토큰이 없거나 조작되었다면 에러(401, 403) 반환
